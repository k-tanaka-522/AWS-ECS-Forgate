# ECSForgate プロジェクト 要件仕様書

## 1. プロジェクト概要

### 1.1 背景と目的
本プロジェクトは、マイクロサービスアーキテクチャを採用したシステムにおける内部APIサービスのサンプル実装として位置づけられています。AWS ECS Fargateを利用したコンテナベースのAPIサービスを構築し、Infrastructure as Code (IaC)によるインフラ管理の実践を目指します。

### 1.2 サービススコープ
- 内部向けマイクロサービスとしての確認記録用API提供
- 他のマイクロサービスからの利用を想定
- JSON形式でのデータ交換
- 複数環境（開発・ステージング・本番）の構築

## 2. 機能要件 (Must)

### 2.1 APIサービス基本機能
- Spring Bootフレームワークを使用した確認記録用API実装
- RESTful APIによるリソース操作の提供
- JSON形式でのデータ送受信
- トランザクション管理の実装

### 2.2 APIエンドポイント
以下のエンドポイントを基本として提供します：

| メソッド | パス             | 機能概要                       |
|---------|-----------------|-------------------------------|
| GET     | /api/records    | 確認記録の一覧取得             |
| GET     | /api/records/{id}| 特定の確認記録の詳細取得      |
| POST    | /api/records    | 新規確認記録の作成             |
| PUT     | /api/records/{id}| 既存確認記録の更新            |
| DELETE  | /api/records/{id}| 確認記録の削除                |
| GET     | /api/health     | APIサービスの死活監視         |

### 2.3 データモデル
確認記録の基本データモデルとして以下を想定します：

```
Record {
  id: UUID             // レコードの一意識別子
  title: String        // 記録タイトル
  content: String      // 記録内容
  status: Enum         // 状態（DRAFT, CONFIRMED, ARCHIVED）
  createdBy: String    // 作成者
  createdAt: Timestamp // 作成日時
  updatedBy: String    // 更新者
  updatedAt: Timestamp // 更新日時
}
```

## 3. 非機能要件

### 3.1 性能要件
- 同時接続数: 最大300ユーザー
- レスポンス時間: 95%のリクエストが500ms以内に応答
- スループット: 1分あたり最大1,000リクエストの処理

### 3.2 可用性要件
- SLA目標: 稼働率99.9%（月間ダウンタイム約43分以内）
- マルチAZ構成（2つのAZ）による可用性確保
- 計画的なメンテナンス時間の設定（月1回、日曜日早朝2〜4時、2時間程度）
- RTO（目標復旧時間）: 30分以内
- RPO（目標復旧時点）: 5分以内

### 3.3 セキュリティ要件
- FISC安全対策基準に準拠したセキュリティ設計
- APIアクセスにはJWTトークン認証方式を採用
- 通信の暗号化（TLS 1.2以上）
- 適切なIAMロールとポリシーの設定による最小権限の原則適用
- セキュリティグループによるアクセス制限
- データの暗号化（保管時および転送時）
- 脆弱性スキャンの定期実施

### 3.4 スケーラビリティ要件
- ECS Fargateによる自動スケーリング
  - CPU使用率70%を超過した場合にスケールアウト
  - 同時リクエスト数に基づいたスケーリングルールの設定
  - 最小2タスク、最大10タスク
- RDSは手動スケールで対応

### 3.5 運用・監視要件
- CloudWatch Logsによるログ収集と分析
  - アプリケーションログ
  - アクセスログ
  - エラーログ
- CloudWatch Alarmsによる異常検知と通知
  - CPU使用率、メモリ使用率、ディスク使用率の監視
  - エラー率のモニタリング
  - レスポンスタイムの監視
- バックアップ戦略
  - RDSの自動バックアップ（日次、保持期間7日）
  - ポイントインタイムリカバリの有効化（5分間隔）
  - 月次のフルバックアップ（保持期間1年）
- X-Rayによるパフォーマンスモニタリングとトレース
- Amazon SNSを使用した障害通知
  - 重大アラートはSMSとEメールで通知
  - 警告レベルはEメールのみで通知

## 4. 技術要件

### 4.1 使用技術
- アプリケーション: Java Spring Boot
- データベース: RDS for PostgreSQL 13以上
- コンテナオーケストレーション: ECS Fargate
- インフラストラクチャ: AWS
  - VPC、サブネット、セキュリティグループ
  - ECS Fargate
  - RDS for PostgreSQL
  - Application Load Balancer
  - API Gateway
  - CloudWatch (Logs, Metrics, Alarms)
  - SNS
  - X-Ray
- IaC: AWS CloudFormation
- リポジトリ: GitHub
- CI/CD: GitHub Actions

### 4.2 環境構成
- 開発環境（dev）
  - 最小リソース構成
  - 開発者向けの柔軟な設定
  - 自動テスト統合
- ステージング環境（stg）
  - 本番相当の構成
  - 性能テスト環境
  - リリース前検証用
- 本番環境（prd）
  - 高可用性構成
  - 適切なリソースサイジング
  - 厳格なセキュリティ設定

## 5. 開発・運用要件

### 5.1 CI/CD要件
- インフラ構築とECSビルドを分離
- GitHubとの連携による自動CI/CD
  - mainブランチへのプッシュで自動ビルド・テスト実行
  - プルリクエストによるコードレビュープロセス
  - ステージング環境へは手動承認後デプロイ
  - 本番環境へはリリースタグ付与後に手動承認でデプロイ
- ビルドプロセスの自動化
  - 単体テスト、統合テスト自動実行
  - コードクオリティチェック（Sonarqube等）
  - セキュリティスキャン
  - コンテナイメージのビルドとタグ付け
- 環境ごとの段階的デプロイ

### 5.2 コスト要件
- サンプル構築のため最低コストでの実装
  - 開発環境: 最小インスタンスサイズの使用
  - ステージング環境: 必要時のみ稼働
  - 本番環境: 適切なサイジングによるコスト最適化
- 不要リソースの自動停止・削除
  - 開発環境の夜間・週末自動停止
  - 未使用リソースの定期的な検出と削除
- コスト監視と最適化
  - AWS Cost Explorerによる定期的なコスト分析
  - 予算アラートの設定
  - リザーブドインスタンスやSavings Planの検討（長期運用時）

### 5.3 ドキュメント要件
- アーキテクチャ設計書
- API仕様書（OpenAPI/Swagger形式）
- インフラ構成図
- 障害復旧手順書
