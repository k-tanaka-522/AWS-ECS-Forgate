# ECSForgate プロジェクト ユースケース定義

## 1. 概要

本ドキュメントでは、ECSForgateプロジェクトのユースケースを定義します。これらのユースケースは、主要な機能要件とシステムの利用シナリオを表現しています。

## 2. アクター定義

システムとの相互作用に関わる主要なアクターを以下に定義します：

- **APIクライアントサービス**: 本APIサービスを利用するマイクロサービス
- **システム管理者**: インフラストラクチャや監視を管理するオペレーター
- **開発者**: アプリケーションやインフラを開発・保守する担当者

## 3. 主要ユースケース

### 3.1 確認記録の管理

#### UC-01: 確認記録の一覧取得
- **アクター**: APIクライアントサービス
- **概要**: 登録されている確認記録の一覧を取得する
- **前提条件**: アクターが有効なJWTトークンを持っている
- **基本フロー**:
  1. アクターはAPIに対して認証トークンを含めたGETリクエストを送信
  2. システムはトークンを検証
  3. システムはデータベースから確認記録の一覧を取得
  4. システムはJSON形式で結果を返却
- **代替フロー**:
  - 認証エラー: システムは401 Unauthorizedエラーを返す
  - システムエラー: システムは500 Internal Server Errorを返す
- **事後条件**: アクターは確認記録の一覧を取得できる

#### UC-02: 特定の確認記録の取得
- **アクター**: APIクライアントサービス
- **概要**: 特定IDの確認記録の詳細を取得する
- **前提条件**: アクターが有効なJWTトークンを持っている
- **基本フロー**:
  1. アクターはAPIに対して認証トークンと記録IDを含めたGETリクエストを送信
  2. システムはトークンを検証
  3. システムはデータベースから指定された記録を取得
  4. システムはJSON形式で結果を返却
- **代替フロー**:
  - 認証エラー: システムは401 Unauthorizedエラーを返す
  - 記録が存在しない: システムは404 Not Foundエラーを返す
  - システムエラー: システムは500 Internal Server Errorを返す
- **事後条件**: アクターは指定した確認記録の詳細を取得できる

#### UC-03: 新規確認記録の作成
- **アクター**: APIクライアントサービス
- **概要**: 新しい確認記録を作成し、システムに保存する
- **前提条件**: アクターが有効なJWTトークンを持っている
- **基本フロー**:
  1. アクターはAPIに対して認証トークンと記録データを含めたPOSTリクエストを送信
  2. システムはトークンを検証
  3. システムはリクエストデータをバリデーション
  4. システムは新しい記録をデータベースに保存
  5. システムは作成された記録のIDと状態をJSON形式で返却
- **代替フロー**:
  - 認証エラー: システムは401 Unauthorizedエラーを返す
  - バリデーションエラー: システムは400 Bad Requestエラーを返す
  - システムエラー: システムは500 Internal Server Errorを返す
- **事後条件**: 新しい確認記録がシステムに保存される

#### UC-04: 確認記録の更新
- **アクター**: APIクライアントサービス
- **概要**: 既存の確認記録の内容を更新する
- **前提条件**: アクターが有効なJWTトークンを持っている、対象の記録が存在する
- **基本フロー**:
  1. アクターはAPIに対して認証トークン、記録ID、更新データを含めたPUTリクエストを送信
  2. システムはトークンを検証
  3. システムはリクエストデータをバリデーション
  4. システムはデータベース内の記録を更新
  5. システムは更新された記録の状態をJSON形式で返却
- **代替フロー**:
  - 認証エラー: システムは401 Unauthorizedエラーを返す
  - 記録が存在しない: システムは404 Not Foundエラーを返す
  - バリデーションエラー: システムは400 Bad Requestエラーを返す
  - システムエラー: システムは500 Internal Server Errorを返す
- **事後条件**: 対象の確認記録が更新される

#### UC-05: 確認記録の削除
- **アクター**: APIクライアントサービス
- **概要**: 特定の確認記録をシステムから削除する
- **前提条件**: アクターが有効なJWTトークンを持っている、対象の記録が存在する
- **基本フロー**:
  1. アクターはAPIに対して認証トークンと記録IDを含めたDELETEリクエストを送信
  2. システムはトークンを検証
  3. システムはデータベースから指定された記録を削除
  4. システムは削除成功のレスポンスを返却
- **代替フロー**:
  - 認証エラー: システムは401 Unauthorizedエラーを返す
  - 記録が存在しない: システムは404 Not Foundエラーを返す
  - システムエラー: システムは500 Internal Server Errorを返す
- **事後条件**: 対象の確認記録がシステムから削除される

### 3.2 システム管理

#### UC-06: システム死活監視
- **アクター**: APIクライアントサービス、システム管理者
- **概要**: システムの稼働状態を確認する
- **前提条件**: なし（認証不要のエンドポイント）
- **基本フロー**:
  1. アクターはAPIの/api/healthエンドポイントにGETリクエストを送信
  2. システムは内部コンポーネント（DB接続など）の状態を確認
  3. システムはヘルスステータスをJSON形式で返却
- **代替フロー**:
  - システムの一部が機能していない: システムは詳細なエラー情報とともに503 Service Unavailableを返す
- **事後条件**: アクターはシステムの稼働状態を把握できる

#### UC-07: パフォーマンスモニタリング
- **アクター**: システム管理者
- **概要**: システムのパフォーマンス指標を監視する
- **前提条件**: CloudWatch監視が設定されている
- **基本フロー**:
  1. アクターはAWSコンソールでCloudWatchダッシュボードにアクセス
  2. システムは設定されたメトリクスを表示
  3. アクターはリアルタイムおよび履歴データを確認
- **代替フロー**:
  - メトリクスデータがない: システムは空のグラフまたはエラーメッセージを表示
- **事後条件**: アクターはシステムのパフォーマンス状況を把握できる

#### UC-08: 障害通知の受信
- **アクター**: システム管理者
- **概要**: システムの異常を通知として受け取る
- **前提条件**: CloudWatchアラームとSNS通知が設定されている
- **基本フロー**:
  1. システムは異常を検出してCloudWatchアラームを発生
  2. システムはSNSを通じて通知を送信
  3. アクターはEメールまたはSMSで通知を受信
- **代替フロー**:
  - 通知配信エラー: システムはCloudWatchログにエラーを記録
- **事後条件**: アクターはシステム異常を迅速に把握できる

### 3.3 開発・デプロイ

#### UC-09: アプリケーションのコード変更
- **アクター**: 開発者
- **概要**: APIコードを変更し、GitHubリポジトリにコミットする
- **前提条件**: 開発者がリポジトリへのプッシュ権限を持っている
- **基本フロー**:
  1. アクターは開発環境でコードを変更
  2. アクターはローカルでテストを実行
  3. アクターは変更をコミットしブランチにプッシュ
  4. アクターはプルリクエストを作成
  5. コードレビュー後、アクターはブランチをマージ
- **代替フロー**:
  - テスト失敗: アクターはテスト結果を確認し、修正を行う
  - コードレビュー指摘: アクターは指摘事項に対応し、再度プッシュする
- **事後条件**: コード変更がマスターブランチに反映される

#### UC-10: インフラストラクチャの変更
- **アクター**: 開発者
- **概要**: CloudFormationテンプレートを変更し、インフラを更新する
- **前提条件**: 開発者がリポジトリへのプッシュ権限およびCloudFormationスタックの更新権限を持っている
- **基本フロー**:
  1. アクターはCloudFormationテンプレートを変更
  2. アクターはCloudFormationテンプレートをバリデーション
  3. アクターは変更をコミットしブランチにプッシュ
  4. アクターはプルリクエストを作成
  5. コードレビュー後、アクターはブランチをマージ
  6. アクターはデプロイスクリプトを実行して開発環境に適用
- **代替フロー**:
  - バリデーションエラー: アクターはエラーを修正し、再度検証
  - デプロイ失敗: アクターは失敗原因を分析し、修正を行う
- **事後条件**: インフラストラクチャの変更が環境に反映される

#### UC-11: 本番環境へのデプロイ
- **アクター**: システム管理者、開発者
- **概要**: ステージング環境で検証済みの変更を本番環境に適用する
- **前提条件**: ステージング環境での検証が完了している、本番環境へのデプロイ権限を持っている
- **基本フロー**:
  1. アクターはリリースブランチを作成しタグ付け
  2. CIパイプラインがビルドとテストを実行
  3. アクターは手動承認を行い本番デプロイを開始
  4. システムは本番環境に変更を適用
  5. システムはデプロイ結果を通知
- **代替フロー**:
  - ビルド/テスト失敗: システムはデプロイを中断し、失敗通知を送信
  - デプロイ失敗: システムは自動ロールバックを試行
- **事後条件**: 変更が本番環境に反映される

## 4. システム境界の定義

本システムは以下のコンポーネントを含みます：

- Spring Bootアプリケーション（API実装）
- ECS Fargateコンテナ環境
- RDS PostgreSQLデータベース
- CloudFormationスタック（インフラ定義）
- CI/CDパイプライン
- 監視・アラート機能

以下のコンポーネントは本システムの外部となります：

- APIを利用するクライアントサービス
- GitHub（コード・IaCリポジトリ）
- 開発者のローカル環境
- AWS管理コンソール
