# AIによるシステム開発プロセスガイド

## 重要な原則
- ユーザーへの質問は一度に1つだけ行い、段階的に進める
- AWS CLIでの直接的な環境変更は避け、常にIaCを通じて変更を管理する
- 設計と製造のフィードバックループを重視し、設計上の課題が見つかった場合は製造よりも設計修正を優先する
- 最小単位での実装とテストを繰り返し行い、早期に問題を発見する
- ユーザーの許可を得ないで更新作業勝手に進めてはいけません。絶対に。

## 1. 開発プロセスの全体像

```
要件定義 → 設計 → 製造(実装) → テスト・デプロイ
    ↑         ↑         |
    |         |         |
    +--------------------+
     (フィードバックループ)
```

## 2. 要件定義フェーズ

### 目的
- サービスの目標、範囲、主要機能を明確化
- 技術要件とビジネス要件の整理
- ステークホルダーとの合意形成

### プラクティス
1. **要件収集**
   - ユーザーとの対話型セッションで詳細な要件を抽出
   - 各要件に優先度を付与（MoSCoW法推奨：Must/Should/Could/Won't）
   - 非機能要件（パフォーマンス、セキュリティ、スケーラビリティ）の明確化

2. **要件ドキュメント化**
   - 要件仕様書の作成（`docs/requirements/`に格納）
   - 要件IDを付与し、トレーサビリティを確保
   - ユースケース図、シーケンス図などの視覚的資料の作成

3. **要件検証**
   - ステークホルダーによるレビュー
   - 曖昧さ、矛盾、欠落の特定と解消
   - 承認プロセスの実施

### 成果物
- 要件仕様書：`docs/requirements/requirements.md`
- ユースケース定義：`docs/requirements/usecases.md`
- 非機能要件定義：`docs/requirements/non_functional.md`
- 用語集：`docs/requirements/glossary.md`

## 3. 設計フェーズ

### 目的
- 要件を満たすアーキテクチャの定義
- コンポーネント間の関係性の明確化
- スケーラビリティ、セキュリティ、可用性の設計方針策定
- 各AWSサービスの構築可能なレベルの詳細設計作成

### プラクティス
1. **アーキテクチャ設計**
   - AWSサービスの選定と構成図の作成
   - ネットワークトポロジーの設計（VPC、サブネット、ルーティング）
   - セキュリティ方針の策定（IAM、セキュリティグループ、暗号化）
   - 各コンポーネントの責任範囲定義

2. **AWSサービス詳細設計**
   - 各AWSサービスごとに以下を含む詳細設計書を作成：
     - **目的と責任範囲**: サービスが担う役割
     - **リソースタイプ**: 使用するAWSリソース（例：AWS::EC2::VPC）
     - **設定パラメータ**: すべての必要なパラメータと設定値
     - **リソース間の依存関係**: 他のAWSリソースとの連携方法
     - **セキュリティ設定**: IAMロール、ポリシー、セキュリティグループなど
     - **監視・アラート設定**: CloudWatchメトリクス、アラームの閾値など
     - **環境差分**: 開発/ステージング/本番環境での差異

3. **AWSサービス詳細設計の例**

   **VPC詳細設計**
   - CIDR設計（VPC、サブネット）
   - アベイラビリティゾーン戦略
   - サブネット階層（パブリック、プライベート、データ層）
   - ルートテーブル設定
   - NACLとセキュリティグループの詳細ルール
   - VPCエンドポイントの設定
   - NAT設定（ゲートウェイ、インスタンス）

   **RDS詳細設計**
   - DBインスタンスタイプとサイズ
   - ストレージ設定（サイズ、IOPS、自動スケーリング）
   - マルチAZ設定
   - パラメータグループの詳細設定
   - バックアップ戦略（自動バックアップ、スナップショット）
   - セキュリティ設定（暗号化、IAM認証）
   - 監視指標とアラート

   **ECS Fargate詳細設計**
   - タスク定義（CPU、メモリ、コンテナ設定）
   - サービス設定（タスク数、デプロイ戦略）
   - ネットワーク設定（サブネット、セキュリティグループ）
   - Auto Scaling設定（ポリシー、ターゲット）
   - IAMロールとポリシー設定
   - ログ設定

   **ALB詳細設計**
   - リスナー設定（プロトコル、ポート）
   - ターゲットグループ設定（ヘルスチェック、ルーティング）
   - セキュリティ設定（TLS、セキュリティグループ）
   - ログ設定とモニタリング
   - WAF連携設定

   **API Gateway詳細設計**
   - エンドポイントタイプと設定
   - リソースとメソッド設計
   - 統合設定（VPCリンク、ALB連携）
   - スロットリングとクォータ設定
   - API認証・認可方式
   - ステージ設定とデプロイ戦略

4. **サービス連携設計**
   - サービス間の依存関係図の作成
   - データフロー図の作成
   - エラーハンドリング戦略の定義
   - イベント連携モデルの定義

5. **設計レビュー**
   - アーキテクチャレビューの実施（AWSのWell-Architected Frameworkに基づく）
   - セキュリティレビューの実施
   - コスト最適化レビューの実施
   - 各AWSサービス詳細設計のレビュー

### 成果物
- アーキテクチャ設計書：`docs/design/architecture.md`
- インフラ構成図：`docs/design/infra_diagram.png`
- 各AWSサービス詳細設計書：`docs/design/components/`
  - VPC詳細設計書：`docs/design/components/vpc-design.md`
  - RDS詳細設計書：`docs/design/components/rds-design.md`
  - ECS Fargate詳細設計書：`docs/design/components/ecs-design.md`
  - ALB詳細設計書：`docs/design/components/alb-design.md`
  - API Gateway詳細設計書：`docs/design/components/apigateway-design.md`
  - その他必要なサービスの詳細設計書
- 環境分離戦略：`docs/design/environments.md`
- IaC設計方針：`docs/design/iac_strategy.md`

## 4. 製造（実装）フェーズ

### 目的
- 設計に基づいたインフラストラクチャの実装
- IaCテンプレートの作成と検証
- コンポーネント間の連携確認

### プラクティス
1. **最小単位での実装**
   - 各AWSサービスを独立したIaCモジュールとして実装
   - 依存関係の少ない順に実装を進める
   - 各モジュールごとにシンプルな検証を行う

2. **継続的なフィードバック**
   - コードレビューの実施
   - 小さな単位でのテストデプロイ
   - 設計との整合性確認

3. **問題発生時の対応**
   - 設計上の問題が発見された場合は製造を一時停止
   - 設計ドキュメントの更新を優先
   - 設計変更の影響範囲を特定してから製造再開

### 成果物
- IaCテンプレート：`iac/cloudformation/templates/`
- 環境別設定：`iac/cloudformation/config/`
- デプロイスクリプト：`iac/cloudformation/scripts/`
- テスト結果：`tests/results/`

## 5. テスト・デプロイフェーズ

### 目的
- 実装の品質確認
- 環境間の安全なデプロイ
- 運用準備

### プラクティス
1. **テスト戦略**
   - IaCテンプレートの静的解析
   - インフラストラクチャテストの実施
   - セキュリティスキャンの実施
   - 負荷テストの実施

2. **段階的デプロイ**
   - 開発環境での初期デプロイ
   - ステージング環境でのユースケーステスト
   - 本番環境への計画的デプロイ

3. **運用準備**
   - モニタリング設定の確認
   - アラート設定の確認
   - バックアップ・復旧手順の確認
   - インシデント対応計画の作成

### 成果物
- テスト報告書：`docs/test/test_report.md`
- デプロイ手順書：`docs/operations/deployment.md`
- 運用手順書：`docs/operations/runbook.md`
- 障害復旧計画：`docs/operations/disaster_recovery.md`

## 6. フィードバックループとプロセス改善

### 製造から設計へのフィードバック
製造フェーズで設計の問題が発見された場合、以下のプロセスを実施します：

1. **問題の記録**
   - `docs/handover.md`に問題の詳細を記録
   - 問題の影響範囲を特定

2. **設計の見直し**
   - 関連する設計ドキュメントの更新
   - 変更による他のコンポーネントへの影響分析

3. **変更管理**
   - 設計変更の承認プロセス実施
   - 変更内容の関係者への周知

4. **製造の再開**
   - 更新された設計に基づく実装の再開
   - 変更の検証

### 設計から要件へのフィードバック
設計フェーズで要件の問題（矛盾、実現不可能な要件など）が発見された場合：

1. **問題の特定と分析**
   - 要件の問題点を明確に記録
   - 代替案や修正案の検討

2. **ステークホルダーとの協議**
   - 問題の説明と影響の共有
   - 要件修正の合意形成

3. **要件の更新**
   - 要件ドキュメントの更新
   - トレーサビリティの維持

## 7. 知識継続性の確保

### 継続ファイルの維持
- `handover.md`: セッション間の引き継ぎ情報を記録
  - 最新の開発状況と進行中のタスク
  - 検討中の課題と決定事項
  - 次回セッションで注目すべきポイント

- `knowledge.md`: プロジェクト全体の知識ベース
  - 重要な設計決定とその理由
  - 採用したパターンやアプローチ
  - トラブルシューティングの履歴と解決策
  - 参考リソースやドキュメントへのリンク

### ファイル運用方法
- 各AIセッション終了時に`handover.md`を更新
- 重要な知見が得られた際に`knowledge.md`を更新
- 新しいセッション開始時にこれらのファイルをAIに提示
- ファイルはリポジトリの`docs/`ディレクトリに配置

## 8. 具体的な実装の進め方

### 依存関係を考慮した実装順序
以下の順序での実装を推奨します：

1. **基盤レイヤー**
   - VPCとネットワーク設定
   - IAMロールとポリシー
   - セキュリティグループ

2. **データレイヤー**
   - RDSインスタンス
   - S3バケット（必要な場合）

3. **アプリケーションレイヤー**
   - ECS Fargate
   - Application Load Balancer

4. **インテグレーションレイヤー**
   - API Gateway
   - VPCエンドポイント

5. **運用レイヤー**
   - CloudWatch設定
   - SNS通知
   - X-Ray設定
   - Amazon Connect

6. **CI/CDレイヤー**
   - CodeCommit
   - デプロイパイプライン

各レイヤーは前のレイヤーの検証が完了してから開始することで、早期に問題を発見し対処できます。

### 最小単位での設計・実装サイクル

1. **小さなスコープで設計**
   - 単一のAWSサービスに焦点を当てた設計
   - 明確な入出力と成功基準の定義

2. **プロトタイプ実装**
   - 設計に基づいた最小限のIaCテンプレート作成
   - ベースとなる機能の実装

3. **検証と評価**
   - テンプレートの検証 (`validate.sh`を使用)
   - 小規模デプロイによる機能確認
   - 設計との整合性チェック

4. **反復と拡張**
   - フィードバックに基づく設計/実装の調整
   - 次の機能追加または洗練

## 9. 環境別の戦略

### 開発環境 (dev)
- 迅速な反復開発を可能にする構成
- 個別コンポーネントの機能検証
- 開発者による即時フィードバック
- コスト最適化のためのリソース制限

### ステージング環境 (stg)
- 本番環境に近い設定での統合テスト
- パフォーマンステストとスケーラビリティ検証
- セキュリティテスト
- 本番リリース前の最終検証

### 本番環境 (prd)
- 高可用性と冗長性の確保
- 厳格なセキュリティ管理
- 包括的なモニタリングとアラート
- 計画的なデプロイプロセス

## 11. サポート戦略

### IaC（Infrastructure as Code）設計
- クラウドプロバイダーに応じた適切なIaCツール（CloudFormation、Terraform等）の選定
- リソース間の依存関係を考慮したモジュール分割
- 環境（dev/stg/prd）ごとのパラメータ化戦略
- セキュリティベストプラクティスに準拠した設計

### クラウドアーキテクチャ設計
- 要件に基づいた最適なクラウドサービスの組み合わせ
- スケーラビリティ、高可用性、セキュリティを考慮した構成
- コスト最適化のためのリソース設計
- Well-Architected Frameworkに準拠した設計

### CI/CD戦略
- プロジェクト特性に合わせたCI/CDツールの選定
- ビルド、テスト、デプロイの各段階の自動化
- 品質ゲートの設定と監視
- ロールバック戦略を含めた安全なデプロイフロー

### コンテナ化戦略
- アプリケーション特性に合わせたコンテナ化アプローチ
- 効率的なDockerfile設計とマルチステージビルド
- コンテナオーケストレーションツール（ECS、Kubernetes等）の選定
- コンテナイメージの効率的なビルド・管理・デプロイ方法

### セキュリティ対策
- 各実装フェーズでのセキュリティチェック
- IAMポリシーの最小権限の原則に基づく設計
- ネットワークセキュリティの強化
- データ保護メカニズムの実装

## 12. まとめ

本ガイドで提案した構造とプロセスにより、以下のメリットが期待できます：

1. **明確なフェーズ分割**
   - 要件定義→設計→製造→テストの流れが明確
   - 各フェーズの責任範囲と成果物が定義されている

2. **効率的なフィードバックループ**
   - 問題発見時の対応プロセスが確立されている
   - 上流工程への迅速なフィードバックが可能

3. **設計優先アプローチ**
   - 実装より設計の正確性を優先
   - 問題の早期発見と低コストでの修正が可能

4. **最小単位での実装と検証**
   - 小さな単位での実装とテストによる早期問題発見
   - 継続的な改善サイクルの確立

このプロセスは、AWSでのプライベート接続APIサービス構築だけでなく、他のクラウドインフラストラクチャプロジェクトにも応用可能です。
