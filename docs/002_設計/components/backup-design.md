# バックアップ設計書

## 1. 概要

### 1.1 目的と責任範囲
本ドキュメントでは、ECSForgateプロジェクトにおけるバックアップに関する設計を定義します。この設計は、データベース、設定、コード、タスク定義などの重要なリソースのバックアップ、保存、復元に関する方針と実装詳細を規定し、障害発生時のデータ保護と迅速なリカバリの実現を担います。

### 1.2 設計のポイント
- 堅牢なバックアップ戦略による障害復旧とデータ保護の実現
- AWS Backupを活用した統合的なバックアップ管理
- 環境ごとに適切なバックアップポリシーと保持期間の定義
- 定期的なバックアップ検証による復元確実性の確保
- FISC安全対策基準に準拠したセキュリティ対策の実装
- コスト効率とデータ保護のバランスを考慮した設計

## 2. バックアップ対象とポリシー

以下のリソースをバックアップ対象とし、AWS Backupを使用して管理します：

| バックアップ対象 | バックアップタイプ | スケジュール | 保持期間 |
|---------------|-----------------|------------|---------| 
| **RDSデータベース** |
| 開発環境 | 自動スナップショット | 毎日 1:00 JST | 7日間 |
| ステージング環境 | 自動スナップショット<br>AWS Backupスナップショット | 毎日 1:00 JST<br>毎週日曜 2:00 JST | 7日間<br>4週間 |
| 本番環境 | 自動スナップショット<br>AWS Backupスナップショット<br>クロスリージョンコピー | 毎日 1:00 JST<br>毎週日曜 2:00 JST<br>毎月1日 3:00 JST | 7日間<br>4週間<br>12ヶ月 |
| **ECSタスク定義** |
| すべての環境 | ECRイメージ | リリース時 | 無期限（最新5バージョン） |
| **CloudFormationスタック** |
| すべての環境 | テンプレート | 変更時 | Gitリポジトリに保存 |
| **設定データ** |
| すべての環境 | Systems Manager Parameter Store | 変更時 | 履歴として保持 |

## 3. RDSバックアップ詳細設計

### 3.1 自動バックアップ設定

RDSインスタンスの自動バックアップを以下のように設定します：

| 設定項目 | 開発環境 | ステージング環境 | 本番環境 |
|---------|---------|----------------|---------| 
| 自動バックアップ | 有効 | 有効 | 有効 |
| バックアップウィンドウ | 01:00-02:00 JST | 01:00-02:00 JST | 01:00-02:00 JST |
| 保持期間 | 7日間 | 7日間 | 7日間 |
| 暗号化 | AWS管理キー | カスタマー管理キー | カスタマー管理キー |

### 3.2 ポイントインタイムリカバリ（PITR）

以下のPITR設定を適用します：

| 環境 | PITR有効/無効 | バックアップ間隔 | 保持期間 |
|-----|--------------|---------------|---------| 
| 開発環境 | 有効 | 5分 | 24時間 |
| ステージング環境 | 有効 | 5分 | 3日間 |
| 本番環境 | 有効 | 5分 | 7日間 |

### 3.3 AWS Backupによる追加バックアップ

AWS Backupを使用して、RDSの追加バックアップを実施します：

| バックアッププラン | スケジュール | コピー先 | ライフサイクル | コールドストレージ移行 |
|-----------------|------------|---------|------------|-------------------| 
| 開発環境週次バックアップ | 毎週日曜 2:00 JST | なし | 2週間で期限切れ | なし |
| ステージング環境週次バックアップ | 毎週日曜 2:00 JST | なし | 4週間で期限切れ | なし |
| 本番環境週次バックアップ | 毎週日曜 2:00 JST | なし | 4週間で期限切れ | なし |
| 本番環境月次バックアップ | 毎月1日 3:00 JST | クロスリージョン (ap-northeast-3) | 12ヶ月で期限切れ | 3ヶ月後 |

### 3.4 バックアップの暗号化

バックアップデータの暗号化設定は以下の通りです：

| 環境 | 暗号化方式 | キー管理 | キーローテーション |
|-----|----------|---------|---------------| 
| 開発環境 | AWS管理KMSキー | AWSサービス管理 | AWSによる自動管理 |
| ステージング環境 | カスタマー管理KMSキー | CloudFormationで管理 | 年次（手動） |
| 本番環境 | カスタマー管理KMSキー | CloudFormationで管理 | 年次（手動） |

## 9. 障害復旧手順

### 9.1 RDSデータベース復旧手順

1. **障害の特定と影響範囲評価**
   - 障害内容の特定（データ破損、インスタンス障害、リージョン障害など）
   - データ損失の可能性と影響範囲の評価
   - バックアップ選択（自動バックアップ、PITR、AWS Backupスナップショット）

2. **復元前の準備**
   - 復元手順と影響のレビュー
   - 必要に応じてアプリケーションの停止またはメンテナンスモード設定
   - 復元計画の承認取得

3. **復元の実施**
   - 選択したバックアップからの復元実行
   - 新しいDBインスタンスとして復元（置き換えではなく）
   - 復元後のデータ検証

4. **アプリケーション接続切り替え**
   - アプリケーション設定の更新（新DBインスタンスへの接続情報）
   - 段階的な接続テスト
   - 本番トラフィックの切り替え

5. **検証と後処理**
   - 復元されたDBの運用状態確認
   - 問題があれば元の状態への巻き戻し
   - インシデント報告と再発防止策の検討

### 9.2 ECSタスク/イメージ復旧手順

1. **障害の特定**
   - タスク定義または実行中タスクの問題特定
   - 問題となるイメージバージョンの特定

2. **復元計画**
   - 以前のバージョンのイメージ選択
   - ロールバックによる影響評価

3. **イメージの復元**
   - ECRから特定バージョンのイメージ取得
   - 必要に応じてタグ付け

4. **タスク更新とデプロイ**
   - タスク定義の更新
   - 新しいタスクのデプロイ
   - 段階的なトラフィック移行（必要に応じて）

5. **検証**
   - 復元されたサービスの動作確認
   - パフォーマンスモニタリング

### 9.3 設定データ復旧手順

1. **設定問題の特定**
   - 問題となっている設定パラメータの特定
   - 正常だった最後のバージョンの特定

2. **復元計画**
   - 古いバージョンのパラメータ取得方法の決定
   - 復元による影響範囲の評価

3. **設定の復元**
   - SSM Parameter StoreやS3から過去バージョンの取得
   - パラメータの更新または再設定

4. **アプリケーション再起動/リデプロイ**
   - 必要に応じてアプリケーションの再起動
   - 設定変更の適用確認

5. **動作確認**
   - 設定が正しく適用されていることの確認
   - アプリケーション機能テスト

### 9.4 CloudFormationスタック復旧手順

1. **問題の特定**
   - スタックの問題箇所の特定
   - 最後の正常テンプレートバージョンの特定

2. **復元計画**
   - GitまたはS3からの古いテンプレート取得
   - スタック更新または再作成の決定

3. **復元実施**
   - 変更セットの作成と確認
   - スタック更新の実行
   - または新しいスタックとしての再作成

4. **リソース検証**
   - すべてのリソースが正しく作成/更新されたことの確認
   - 依存リソースとの連携確認

5. **切り替え**
   - 新しいスタックを作成した場合は、リソース参照の更新
   - DNSやロードバランサーの切り替え（必要に応じて）

## 10. 環境差分

環境ごとのバックアップに関する主な差分は以下の通りです：

| 設定項目 | 開発環境（dev） | ステージング環境（stg） | 本番環境（prd） |
|---------|--------------|-------------------|--------------| 
| バックアップ保持期間 | 短期間（7-14日） | 中期間（1-4週間） | 長期間（最大1年） |
| バックアップ頻度 | 低頻度（日次のみ） | 中頻度（日次+週次） | 高頻度（日次+週次+月次） |
| リージョン間コピー | なし | なし | あり（災害対策） |
| 暗号化キー | AWS管理キー | カスタマー管理キー | カスタマー管理キー |
| 復元テスト | テスト対象 | 必要時のみ | 本番リカバリ演習用 |
| バックアップ通知 | メールのみ | メールのみ | メール + SMS |

## 11. CloudFormationリソース定義

バックアップのためのCloudFormationリソースタイプと主要パラメータを以下に示します：

```yaml
Resources:
  # バックアップボールト
  BackupVault:
    Type: AWS::Backup::BackupVault
    Properties:
      BackupVaultName: !Sub "ecsforgate-${EnvironmentName}-vault"
      EncryptionKeyArn: !If [UseCustomKey, !GetAtt BackupKmsKey.Arn, !Ref "AWS::NoValue"]
      
  # バックアッププラン
  DailyBackupPlan:
    Type: AWS::Backup::BackupPlan
    Properties:
      BackupPlan:
        BackupPlanName: !Sub "ECSForgate-${EnvironmentName}-DailyBackup"
        BackupPlanRule:
          - RuleName: "DailyBackups"
            TargetBackupVault: !Ref BackupVault
            ScheduleExpression: "cron(0 1 * * ? *)"
            StartWindowMinutes: 60
            CompletionWindowMinutes: 180
            Lifecycle:
              DeleteAfterDays: !FindInMap [EnvironmentMap, !Ref EnvironmentName, BackupRetention]
        
  # リソース割り当て
  BackupResourceAssignment:
    Type: AWS::Backup::BackupSelection
    Properties:
      BackupPlanId: !Ref DailyBackupPlan
      BackupSelection:
        SelectionName: !Sub "ECSForgate-${EnvironmentName}-Resources"
        IamRoleArn: !GetAtt BackupRole.Arn
        Resources:
          - !Sub "arn:aws:rds:${AWS::Region}:${AWS::AccountId}:db:${DBInstanceName}"
          
  # ECRリポジトリ
  ContainerRepository:
    Type: AWS::ECR::Repository
    Properties:
      RepositoryName: !Sub "ecsforgate-${EnvironmentName}"
      ImageScanningConfiguration:
        ScanOnPush: true
      LifecyclePolicy:
        LifecyclePolicyText: !If
          - IsProd
          - |
            {
              "rules": []
            }
          - |
            {
              "rules": [
                {
                  "rulePriority": 1,
                  "description": "Keep only 10 images",
                  "selection": {
                    "tagStatus": "any",
                    "countType": "imageCountMoreThan",
                    "countNumber": 10
                  },
                  "action": {
                    "type": "expire"
                  }
                }
              ]
            }
      RepositoryPolicyText:
        Version: "2012-10-17"
        Statement:
          - Sid: AllowPull
            Effect: Allow
            Principal:
              Service: ecs-tasks.amazonaws.com
            Action:
              - ecr:GetDownloadUrlForLayer
              - ecr:BatchGetImage
              - ecr:BatchCheckLayerAvailability
```

## 12. 考慮すべきリスクと対策

| リスク | 影響度 | 対策 |
|-------|-------|------|
| バックアップ失敗 | 高 | 監視強化、定期的な復元テスト、バックアップ成功通知 |
| バックアップからの復元時間遅延 | 高 | 定期的な復元テスト実施、復元手順の最適化と自動化 |
| 暗号化キー紛失 | 高 | キー管理プロセスの厳格化、ローテーションスケジュール管理 |
| バックアップコストの増大 | 中 | 保持期間の最適化、ライフサイクルポリシーの適用 |
| クロスリージョンレプリケーション遅延 | 中 | 監視強化、定期的な複製状態確認 |
| 復元テスト不足 | 高 | テストスケジュールの厳格化、自動化テストの導入 |

## 13. 運用手順

### 13.1 日次運用タスク

- **バックアップ状態確認**: 自動バックアップの成功確認
- **アラート確認**: バックアップ関連のアラート状態確認
- **ストレージ使用状況**: バックアップストレージの使用量確認

### 13.2 週次運用タスク

- **AWS Backupジョブ確認**: 週次バックアップジョブの成功確認
- **ECRイメージスキャン結果確認**: イメージ脆弱性スキャン結果の確認
- **バックアップライフサイクル管理**: 期限切れバックアップの確認

### 13.3 月次運用タスク

- **バックアップ復元テスト**: 開発環境での復元テスト実施
- **クロスリージョンコピー確認**: 別リージョンへのバックアップコピー確認
- **バックアップ設定レビュー**: バックアップポリシーと保持期間の見直し

### 13.4 四半期/半期運用タスク

- **総合リカバリテスト**: 複数コンポーネントの復元テスト
- **災害復旧演習**: クロスリージョン復元を含む総合演習
- **バックアップ戦略見直し**: コスト効率と保護レベルのバランス評価
