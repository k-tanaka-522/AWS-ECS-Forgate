# RDS詳細設計書

## 1. 概要

### 1.1 目的と責任範囲
本ドキュメントでは、ECSForgateプロジェクトにおけるRDS（Relational Database Service）の詳細設計を定義します。RDSはプロジェクトのデータ永続化層として機能し、確認記録データの管理と保存を担当します。

### 1.2 設計のポイント
- FISC安全対策基準に準拠したセキュアなデータベース環境の構築
- 高可用性と耐障害性を確保するためのマルチAZ構成
- 効率的なバックアップと復元戦略の実装
- パフォーマンスとコストのバランスを考慮したインスタンスサイジング
- 環境ごとの要件に適した構成の定義

## 2. データベースエンジンの選定

### 2.1 データベースエンジンの選択理由
PostgreSQL 13.x を選択した理由は以下の通りです：

- **信頼性**: オープンソースのエンタープライズ級データベースとして広く採用されている
- **機能性**: JSON型サポート、高度なインデックス機能、強力なトランザクション管理
- **パフォーマンス**: 効率的なクエリ実行計画と最適化機能
- **セキュリティ**: 堅牢なセキュリティ機能とRDSでの暗号化サポート
- **コスト効率**: 商用ライセンスが不要でOSSとしてサポートされている

### 2.2 バージョン選定
PostgreSQL 13.x を選択する理由：

- 安定性と信頼性が実証されたメジャーバージョン
- AWSによる長期サポート対象
- 最新のセキュリティパッチが適用可能
- 拡張性とパフォーマンスの改善を含む

## 3. インスタンスタイプとサイジング

### 3.1 環境別インスタンスタイプ

| 環境 | インスタンスタイプ | vCPU | メモリ | ストレージタイプ | 初期サイズ | 自動スケーリング | マルチAZ |
|-----|-----------------|------|-------|--------------|----------|-------------|---------|
| 開発環境（dev） | db.t3.small | 2 | 2 GiB | gp2 | 20 GB | 無効 | 無効 |
| ステージング環境（stg） | db.t3.medium | 2 | 4 GiB | gp2 | 50 GB | 有効（最大100GB） | 有効 |
| 本番環境（prd） | db.t3.large | 2 | 8 GiB | gp3 | 100 GB | 有効（最大200GB） | 有効 |

### 3.2 サイジングの考慮点

- **開発環境**: コスト最適化を優先し、最小構成で十分なリソースを確保
- **ステージング環境**: 本番相当のテストが可能な構成を確保しつつ、コストも考慮
- **本番環境**: パフォーマンスと安定性を優先した適切なサイジングを実施

### 3.3 ストレージ設計

- **タイプ**: 
  - 開発/ステージング: GP2（汎用SSD）
  - 本番: GP3（パフォーマンスチューニング可能な次世代SSD）

- **初期サイズ**:
  - 開発: 20GB（最小サイズ）
  - ステージング: 50GB
  - 本番: 100GB

- **自動スケーリング設定**:
  - 開発: 無効
  - ステージング: 有効（最大100GB、スケールアップ閾値: 使用率80%）
  - 本番: 有効（最大200GB、スケールアップ閾値: 使用率80%）

- **IOPS設定**:
  - 開発/ステージング: GP2の標準IOPS（サイズに応じた自動割り当て）
  - 本番: GP3の基本IOPS 3,000に加え、追加IOPS 1,000（合計4,000 IOPS）

## 4. マルチAZ設定

### 4.1 環境別マルチAZ構成

| 環境 | マルチAZ | 導入方法 | プライマリAZ | スタンバイAZ |
|-----|---------|---------|-------------|------------|
| 開発環境（dev） | 無効 | - | ap-northeast-1a | - |
| ステージング環境（stg） | 有効 | 標準のRDSマルチAZ | ap-northeast-1a | ap-northeast-1c |
| 本番環境（prd） | 有効 | 標準のRDSマルチAZ | ap-northeast-1a | ap-northeast-1c |

### 4.2 フェイルオーバーの考慮点

- **フェイルオーバー時間**: 一般的に60〜120秒程度
- **アプリケーション側の対応**: 接続エラー時のリトライロジックを実装
- **CNAMEの自動更新**: RDSエンドポイントは変わらないため、DNS更新は不要
- **監視**: CloudWatchでフェイルオーバーイベントを監視し、アラート設定

### 4.3 読み取りレプリカ

- 現時点では読み取りレプリカは導入せず
- 将来的な拡張オプションとして検討可能
- 実装する場合は、クロスAZでの配置を検討

## 5. パラメータグループの詳細設定

### 5.1 カスタムパラメータグループ

PostgreSQL用のカスタムパラメータグループを作成し、以下の設定を適用します：

| パラメータ名 | 値 | 説明 |
|------------|-----|-----|
| max_connections | 開発：100<br>ステージング：200<br>本番：300 | 同時接続数の最大値 |
| shared_buffers | 開発：256MB<br>ステージング：1GB<br>本番：2GB | PostgreSQLのメモリバッファサイズ |
| effective_cache_size | 開発：768MB<br>ステージング：3GB<br>本番：6GB | クエリプランナーが想定するキャッシュサイズ |
| work_mem | 開発：4MB<br>ステージング：8MB<br>本番：16MB | ソートなどの操作に使用するメモリ量 |
| maintenance_work_mem | 開発：64MB<br>ステージング：128MB<br>本番：256MB | VACUUM、INDEX作成などのメンテナンス操作用メモリ |
| log_min_duration_statement | 開発：0（すべて）<br>ステージング：100ms<br>本番：300ms | 実行時間がこの値以上のクエリをログに記録 |
| log_statement | 開発：all<br>ステージング：mod<br>本番：mod | ログに記録するSQL文の種類 |
| log_connections | true | 接続試行をログに記録 |
| log_disconnections | true | セッション終了時の詳細をログに記録 |
| idle_in_transaction_session_timeout | 3600000 (1時間) | アイドル状態のトランザクションをタイムアウトさせる時間（ミリ秒） |
| statement_timeout | 開発：60000 (1分)<br>ステージング：300000 (5分)<br>本番：600000 (10分) | SQLステートメントの最大実行時間（ミリ秒） |

### 5.2 パラメータグループの適用方法

- 環境ごとに専用のパラメータグループを作成
- CloudFormationでパラメータグループを定義し、RDSインスタンスにアタッチ
- パラメータ変更後の反映方法（静的/動的）に応じた適用計画

## 7. セキュリティ設定

### 7.1 暗号化

| 環境 | 保存時の暗号化 | 暗号化キー管理 | 転送時の暗号化 | 
|-----|-------------|--------------|-------------|
| 開発環境（dev） | 有効 | AWS管理のKMSキー | 有効（TLS 1.2） |
| ステージング環境（stg） | 有効 | カスタマー管理のKMSキー | 有効（TLS 1.2） |
| 本番環境（prd） | 有効 | カスタマー管理のKMSキー | 有効（TLS 1.2） |

### 7.2 ネットワークセキュリティ

- **サブネット配置**: プライベートデータサブネット内にRDSインスタンスを配置
- **セキュリティグループ設定**:
  - インバウンドルール: PostgreSQLポート（5432）へのアクセスをアプリケーション層のセキュリティグループからのみ許可
  - アウトバウンドルール: すべてのトラフィックを許可（デフォルト）

### 7.3 アクセス制御

- **IAM DB認証**: 有効（開発環境から）
- **マスターユーザー管理**:
  - 環境ごとに個別のマスターユーザー認証情報
  - AWS Secrets Managerでの認証情報管理
- **アプリケーション用ユーザー**:
  - 最小権限の原則に基づいた権限設定
  - 環境ごとに個別のユーザー/パスワード

### 7.4 監査とログ記録

- **PostgreSQLログ**: CloudWatch Logsに転送
- **ログ保持期間**:
  - 開発環境: 14日間
  - ステージング環境: 30日間
  - 本番環境: 90日間
- **アクティビティログ**:
  - 接続ログの有効化
  - DDL変更のログ記録
  - スロークエリのログ記録

## 10. データベーススキーマと初期データ

### 10.1 データベーススキーマ管理

- **マイグレーションツール**: Flyway（Java向けDBマイグレーションツール）
- **バージョン管理**: マイグレーションスクリプトはGitリポジトリで管理
- **マイグレーション実行**: CI/CDパイプラインの一部として自動実行

### 10.2 初期スキーマ

```sql
-- 確認記録テーブル
CREATE TABLE records (
  id UUID PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  content TEXT NOT NULL,
  status VARCHAR(20) NOT NULL CHECK (status IN ('DRAFT', 'CONFIRMED', 'ARCHIVED')),
  created_by VARCHAR(100) NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL,
  updated_by VARCHAR(100) NOT NULL,
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL
);

-- インデックス作成
CREATE INDEX idx_records_status ON records(status);
CREATE INDEX idx_records_created_at ON records(created_at);
CREATE INDEX idx_records_updated_at ON records(updated_at);
```

### 11.2 接続文字列管理

- **接続情報の保管**: AWS Secrets Managerを使用

- **ローテーション**: シークレットの自動ローテーションを設定

### 11.3 接続障害対策

- コネクションプールの正常性チェック
- 接続エラー時の適切なリトライ処理の実装
- フェイルオーバー発生時のプール再構築メカニズム

## 12. パフォーマンスチューニング

### 12.1 インデックス戦略

- 主キー（UUID）とは別に検索頻度の高いカラムにインデックスを作成
- 複合インデックスは慎重に検討し、クエリパターンに基づいて設計
- 大量データ操作時のインデックスの一時的無効化を検討

### 12.2 クエリ最適化

- 実行計画の定期的な確認と最適化
- スロークエリのログ分析と改善
- N+1問題を回避するための適切なフェッチ戦略

### 12.3 パフォーマンス監視とチューニング

- CloudWatch Performance Insightsの活用
- スロークエリのトラッキングと最適化
- 定期的なExplain Analyze実行によるクエリプランの評価

## 14. CloudFormationリソース定義

RDS構成のためのCloudFormationリソースタイプと主要パラメータを以下に示します：

### 14.1 主要リソースタイプ
- **AWS::RDS::DBInstance**: PostgreSQLデータベースインスタンスを定義
- **AWS::RDS::DBSubnetGroup**: データベースサブネットグループを定義
- **AWS::RDS::DBParameterGroup**: データベースパラメータグループを定義
- **AWS::SecretsManager::Secret**: データベース認証情報を管理
- **AWS::EC2::SecurityGroup**: データベースアクセス用のセキュリティグループを定義
- **AWS::CloudWatch::Alarm**: データベースメトリクスのアラームを定義

### 14.2 主要パラメータ例

```yaml
Parameters:
  EnvironmentName:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - stg
      - prd
    Description: 環境名（dev/stg/prd）

    Type: String

      - db.t3.small
```

## 15. 考慮すべきリスクと対策

| リスク | 影響度 | 対策 |
|-------|-------|------|
| ストレージ容量枯渇 | 高 | 自動スケーリングの有効化、使用量監視アラート設定 |
| パフォーマンス低下 | 中 | 適切なインデックス設計、定期的なパフォーマンスモニタリング |
| データ損失 | 高 | 適切なバックアップ計画、PITRの有効化、定期的な復元テスト |
| セキュリティ侵害 | 高 | プライベートサブネット配置、最小権限の適用、暗号化の実施 |
| 接続の枯渇 | 中 | 適切なコネクションプール設定、高い接続数のアラート設定 |
| フェイルオーバー失敗 | 高 | 定期的なフェイルオーバーテスト、アプリケーションのリトライ機能 |

## 16. 設計検証方法

本RDS設計の妥当性を確認するために、以下の検証を実施します：

1. **パフォーマンステスト**:
   - 想定される最大負荷でのスループットテスト
   - パフォーマンスインサイトを使用したボトルネック分析

2. **耐障害性テスト**:
   - フェイルオーバーテスト（マルチAZ構成の場合）
   - アプリケーションの接続復旧機能の検証

3. **バックアップと復元テスト**:
   - 自動バックアップからの復元テスト
   - ポイントインタイムリカバリの有効性確認

4. **セキュリティテスト**:
   - 不正アクセス試行に対する防御確認
   - 暗号化設定の検証
   - IAM認証の検証（有効化されている場合）
