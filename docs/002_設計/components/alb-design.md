## 1. 概要

### 1.1 目的と責任範囲
本ドキュメントでは、ECSForgateプロジェクトにおけるApplication Load Balancer（ALB）の詳細設計を定義します。ALBはプロジェクトのAPIトラフィック分散層として機能し、高可用性確保とECSコンテナへのリクエストルーティングを担当します。

### 1.2 設計のポイント
- 内部ALBによるプライベートAPIトラフィックの効率的なルーティング
- マルチAZ配置による高可用性の確保
- ECS Fargateタスクへの適切なヘルスチェックと負荷分散
- セキュリティを考慮したアクセス制御設計
- 将来の拡張性を見据えたリスナー・ルール設計

## 2. ALB基本設計

### 2.1 ALBタイプと命名規則

| 環境 | ALB名 | タイプ | スキーム | 
|-----|------|------|---------|
| 開発環境（dev） | ecsforgate-dev-alb | Application | internal |
| ステージング環境（stg） | ecsforgate-stg-alb | Application | internal |
| 本番環境（prd） | ecsforgate-prd-alb | Application | internal |

### 2.2 基本スペック

- **タイプ**: Application Load Balancer（レイヤー7）
- **スキーム**: 内部（インターネットからアクセス不可）
- **IPアドレスタイプ**: IPv4
- **ロードバランサー世代**: 最新世代（アクセスログに追加フィールド対応）
- **削除保護**: 開発環境：無効、ステージング/本番環境：有効

## 3. ネットワーク構成

### 3.1 配置サブネット

| 環境 | サブネット | アベイラビリティゾーン |
|-----|----------|----------------------|
| 開発環境 | PrivateAppSubnetA | ap-northeast-1a |
| ステージング環境 | PrivateAppSubnetA<br>PrivateAppSubnetB | ap-northeast-1a<br>ap-northeast-1c |
| 本番環境 | PrivateAppSubnetA<br>PrivateAppSubnetB | ap-northeast-1a<br>ap-northeast-1c |

### 3.2 セキュリティグループ設定

ALB用セキュリティグループは以下のように設定します：

- **名前**: `ecsforgate-{env}-alb-sg`
- **説明**: Security group for ECSForgate ALB
- **VPC**: プロジェクトVPC

**インバウンドルール**:

| タイプ | プロトコル | ポート範囲 | ソース | 説明 |
|------|----------|---------|-------|------|
| HTTP | TCP | 80 | 10.0.0.0/16 | 同一VPC内からのHTTPアクセス |
| HTTPS | TCP | 443 | 10.0.0.0/16 | 同一VPC内からのHTTPSアクセス（将来用） |

**アウトバウンドルール**:

| タイプ | プロトコル | ポート範囲 | 送信先 | 説明 |
|------|----------|---------|-------|------|
| カスタムTCP | TCP | 8080 | {ECS SG ID} | ECSコンテナへのアクセス |

### 3.3 VPC設定

- **VPC ID**: プロジェクトVPC ID
- **サブネット**: プライベートアプリケーションサブネット（各AZ）

## 5. ターゲットグループ設定

### 5.1 ターゲットグループの基本情報

| 環境 | ターゲットグループ名 | プロトコル | ポート | ターゲットタイプ | VPC |
|-----|------------------|----------|------|--------------|-----|
| 開発環境 | ecsforgate-dev-tg | HTTP | 8080 | ip | プロジェクトVPC |
| ステージング環境 | ecsforgate-stg-tg | HTTP | 8080 | ip | プロジェクトVPC |
| 本番環境 | ecsforgate-prd-tg | HTTP | 8080 | ip | プロジェクトVPC |

### 5.2 ターゲット登録方法

- **ターゲットタイプ**: IP（ECS Fargateの場合は必須）
- **登録方法**: ECSサービス定義に基づく動的登録
- **Target構成**: 各ECS Fargateタスクの内部IPアドレス:8080

### 5.3 ヘルスチェック設定

| パラメータ | 値 | 説明 |
|----------|-----|------|
| プロトコル | HTTP | ヘルスチェックに使用するプロトコル |
| パス | /api/health | ヘルスチェックに使用するURLパス |
| ポート | 8080 | ヘルスチェックに使用するポート（trafficと同じポート） |
| 正常しきい値 | 3 | ターゲットを正常と見なすまでに必要な連続正常ヘルスチェック |
| 非正常しきい値 | 2 | ターゲットを異常と見なすまでに必要な連続異常ヘルスチェック |
| タイムアウト | 5秒 | ヘルスチェックのタイムアウト時間 |
| 間隔 | 30秒 | ヘルスチェックの実行間隔 |
| 成功コード | 200-299 | 正常と見なすHTTPステータスコード |

### 5.4 詳細設定

| パラメータ | 値 | 説明 |
|----------|-----|------|
| ロードバランスアルゴリズム | ラウンドロビン | ターゲット選択アルゴリズム |
| スロースタートの持続時間 | 0秒（開発）<br>30秒（ステージング）<br>30秒（本番） | 新規ターゲットへのトラフィック増加率調整 |
| スティッキーセッション | 無効 | セッション維持は不要（ステートレス設計） |
| 接続解除 | 無効 | 登録解除時の接続ドレイン待機なし |
| Target異常値の登録解除 | 無効 | 異常ターゲットの自動登録解除しない（ECSが管理） |

### 7.1 ロードバランサー属性

| 属性 | 値 | 説明 |

| アイドルタイムアウト | 60秒 | 接続アイドル状態の最大許容時間 |

| 属性 | 値 | 説明 |

## 11. CloudFormationリソース定義

ALB構成のためのCloudFormationリソースタイプと主要パラメータを以下に示します：

### 11.1 主要リソースタイプ
- **AWS::ElasticLoadBalancingV2::LoadBalancer**: ALBを定義
- **AWS::ElasticLoadBalancingV2::Listener**: ALBリスナーを定義
- **AWS::ElasticLoadBalancingV2::ListenerRule**: リスナールールを定義
- **AWS::ElasticLoadBalancingV2::TargetGroup**: ターゲットグループを定義
- **AWS::EC2::SecurityGroup**: ALBセキュリティグループを定義
- **AWS::S3::Bucket**: ALBログバケットを定義
- **AWS::CloudWatch::Alarm**: ALBメトリクスアラームを定義

### 11.2 主要パラメータ例

```yaml
Parameters:
  EnvironmentName:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - stg
      - prd
    Description: 環境名（dev/stg/prd）

  VpcId:
    Type: AWS::EC2::VPC::Id
    Description: VPC ID

  PrivateAppSubnetA:
    Type: AWS::EC2::Subnet::Id
    Description: アプリケーション層プライベートサブネットA

  PrivateAppSubnetB:
    Type: AWS::EC2::Subnet::Id
    Description: アプリケーション層プライベートサブネットB

  MultiAzEnabled:
    Type: String
    Default: false
    AllowedValues:
      - true
      - false
    Description: マルチAZ配置を有効にするかどうか

  DeletionProtection:
    Type: String
    Default: false
    AllowedValues:
      - true
      - false
    Description: 削除保護を有効にするかどうか

  LogRetentionDays:
    Type: Number
    Default: 14
    AllowedValues:
      - 7
      - 14
      - 30
      - 60
      - 90
      - 180
      - 365
    Description: ログの保持期間（日）
```

## 12. 考慮すべきリスクと対策

| リスク | 影響度 | 対策 |
|-------|-------|------|
| 単一障害点リスク | 高 | マルチAZ構成、ターゲットの適切な分散 |
| トラフィック急増 | 中 | Auto Scalingの設定、バッファの確保、スロットリング検討 |
| 誤設定 | 高 | IaCによる構成管理、変更のレビュープロセス |
| セキュリティリスク | 高 | 適切なセキュリティグループ、定期的な脆弱性スキャン |
| ヘルスチェック不備 | 中 | 適切なヘルスチェックパス、間隔、閾値設定 |
| コスト増加 | 低 | メトリクス監視、不要リソースの削除 |

## 13. 設計検証方法

本ALB設計の妥当性を確認するために、以下の検証を実施します：

### 13.1 機能検証
- 基本的なリクエストルーティングの確認
- ヘルスチェックの動作確認
- リスナーとリスナールールの検証
- セキュリティグループの効果確認

### 13.2 性能検証
- ベースラインパフォーマンスの測定
- 負荷テストによる限界値確認
- レスポンスタイムの測定
- 連続アクセス時の安定性確認

### 13.3 可用性検証
- AZフェイルオーバーテスト（マルチAZ環境）
- 異常ターゲット検出と切り離し動作確認
- サービス再起動時の影響確認
- メンテナンス時のダウンタイム確認

### 13.4 障害対応検証
- 主要コンポーネント障害時の動作確認
- アラート通知の検証
- 異常時のログ記録確認
- 復旧手順の有効性検証

## 14. 運用考慮点

### 14.1 通常運用時の考慮点
- ALBメトリクスとログの定期確認
- ヘルスチェック状態の監視
- アクセスパターンの分析
- 負荷とキャパシティの予測

### 14.2 障害発生時の考慮点
- ALBログとCloudWatchメトリクスによる異常検知
- ECSログとの相関分析
- 一時的なトラフィック迂回策
- 障害報告と原因分析手順

### 14.3 メンテナンス時の考慮点
- 設定変更の適用タイミング
- 証明書ローテーション手順（HTTPS対応時）
- ロギング設定の定期見直し
- パフォーマンスチューニングの実施
