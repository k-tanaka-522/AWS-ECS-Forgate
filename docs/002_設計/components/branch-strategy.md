# ブランチ戦略とCI/CDパイプラインの関連付け

## 概要

本ドキュメントでは、ECSForgateプロジェクトにおけるGitブランチ戦略とCI/CDパイプラインの関連付けについて定義します。GitFlowをベースとしたブランチ戦略を採用し、各ブランチと対応する環境、CI/CDパイプラインの動作を明確にします。

## 1. ブランチ戦略

GitFlowをベースとした以下のブランチ戦略を採用します：

| ブランチ | 用途 | デプロイ先環境 | 保護設定 |
|---------|------|-------------|----------|
| `main` | 本番リリース用 | 本番環境(prd) | 保護あり（PR必須、承認必須、CI合格必須） |
| `develop` | 開発統合用 | 開発環境(dev) | 保護あり（PR必須、CI合格必須） |
| `feature/*` | 新機能開発用 | なし（PRで`develop`へ） | なし |
| `release/*` | リリース準備用 | ステージング環境(stg) | 保護あり（PR必須） |
| `hotfix/*` | 緊急修正用 | 本番環境(修正後) | なし |

## 2. 環境とブランチの対応

環境ごとに監視対象とするブランチと、デプロイの挙動を定義します：

### 2.1 開発環境 (dev)

- **監視ブランチ**: `develop`
- **デプロイトリガー**: ブランチへのプッシュ
- **承認フロー**: 不要（自動デプロイ）
- **デプロイ方式**: ローリング更新
- **ロールバック**: 自動（ヘルスチェック失敗時）

### 2.2 ステージング環境 (stg)

- **監視ブランチ**: `release/*`
- **デプロイトリガー**: ブランチへのプッシュ
- **承認フロー**: オプション（QA承認）
- **デプロイ方式**: ブルー/グリーン
- **ロールバック**: 自動（ヘルスチェック失敗時）

### 2.3 本番環境 (prd)

- **監視ブランチ**: `main`
- **デプロイトリガー**: ブランチへのプッシュ
- **承認フロー**: 必須（運用管理者承認）
- **デプロイ方式**: ブルー/グリーン
- **ロールバック**: 自動+手動オプション

## 3. CI/CDパイプラインの設定

各環境に対応するパイプラインの設定を定義します：

### 3.1 CI/CDパイプラインのステージ構成

すべての環境のパイプラインは以下のステージで構成されます：

1. **Source**: GitHubからソースコードを取得
2. **Build**: アプリケーションのビルドとDockerイメージの作成
3. **Approval**: 承認ステップ（環境によって異なる）
4. **Deploy**: CloudFormationによるインフラおよびアプリケーションのデプロイ

### 3.2 環境別のCI/CDパイプライン設定

| ステージ | 開発環境 (dev) | ステージング環境 (stg) | 本番環境 (prd) |
|---------|--------------|-------------------|--------------| 
| Source | GitHub (develop) | GitHub (release/*) | GitHub (main) |
| Build | ecsforgate-dev-build | ecsforgate-stg-build | ecsforgate-prd-build |
| Test | 基本テスト | 拡張テスト + セキュリティスキャン | 拡張テスト + セキュリティスキャン |
| Approval | 不要 | オプション（QA承認） | 必須（本番承認者） |
| Deploy | CloudFormation + CodeDeploy | CloudFormation + CodeDeploy | CloudFormation + CodeDeploy |

## 4. ワークフロー例

### 4.1 新機能開発

1. `feature/new-feature`ブランチを`develop`から作成
2. 開発作業を実施
3. 完了したら`develop`へのPRを作成
4. コードレビュー、自動テスト実行
5. PRマージ後、自動的に開発環境へデプロイ

### 4.2 リリース準備

1. `release/v1.0.0`ブランチを`develop`から作成
2. 必要に応じてバグ修正等を実施
3. `release/v1.0.0`へのプッシュでステージング環境へ自動デプロイ
4. QAテスト実施
5. 問題なければ`main`へのPRを作成
6. レビュー、承認後にマージ
7. `main`へのマージで本番環境へのデプロイパイプラインが起動
8. 運用管理者の承認後、本番環境へデプロイ

### 4.3 緊急修正（ホットフィックス）

1. `hotfix/critical-fix`ブランチを`main`から作成
2. 修正作業を実施
3. 完了したら`main`へのPRを作成
4. レビュー、承認後にマージ
5. `main`へのマージで本番環境へのデプロイパイプラインが起動
6. 運用管理者の承認後、本番環境へデプロイ
7. その後、`develop`へもマージして修正を反映

## 5. GitHubリポジトリ設定

### 5.1 ブランチ保護ルール

以下のブランチ保護ルールを設定します：

- **main**:
  - PRによるマージ必須
  - 承認レビュー必須（最低1件）
  - ステータスチェック（CI）必須
  - 管理者による強制プッシュの禁止

- **develop**:
  - PRによるマージ必須
  - ステータスチェック（CI）必須

- **release/**:
  - PRによるマージ必須

### 5.2 Webhookの設定

CI/CDパイプラインを自動的に起動するためのWebhookを設定します：

- **トリガーイベント**: プッシュイベント
- **ブランチフィルター**:
  - `main` → 本番環境パイプライン
  - `develop` → 開発環境パイプライン
  - `release/*` → ステージング環境パイプライン

## 6. 考慮すべき点と対策

- **並行開発**: 複数の`feature`ブランチが同時進行する場合の競合解決
- **リリース間隔**: 定期的なリリースサイクルの確立
- **ホットフィックス後の同期**: 本番修正が`develop`に確実に反映されるフロー
- **タグ付け**: リリースバージョンの明確化のためのタグ戦略
- **デプロイ通知**: 各ステージのデプロイ結果をチーム全体への通知方法
