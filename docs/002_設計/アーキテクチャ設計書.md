# ECSForgate プロジェクト アーキテクチャ設計書

## 1. アーキテクチャ概要

### 1.1 システム概要
本プロジェクトは、AWS ECS Fargateを利用したコンテナベースのRESTful APIサービスを提供するマイクロサービスアーキテクチャを実装します。Spring Bootフレームワークを使用した確認記録用APIを、AWSの各種マネージドサービスを組み合わせて構築します。

### 1.2 アーキテクチャの特徴と設計方針

#### 特徴
- マイクロサービスアーキテクチャによる独立したAPIサービス
- コンテナベースのデプロイとサーバーレス運用（ECS Fargate）
- プライベートVPC内でのセキュアなAPI提供
- Infrastructure as Code（IaC）による環境構築の自動化
- マルチAZ構成による高可用性
- 自動スケーリングによる効率的なリソース利用

#### 設計方針
- セキュリティ優先：FISC安全対策基準に準拠したセキュリティ設計
- 高可用性：マルチAZ構成とフェイルオーバーによる99.9%の稼働率確保
- 運用効率化：モニタリング、ログ管理、自動アラートの整備
- コスト最適化：必要なリソースを必要な時に利用する設計
- スケーラビリティ：負荷に応じた自動スケーリング機能の実装
- 保守性：標準的なマイクロサービスパターンの採用とIaCによる構成管理

## 3. コンポーネント構成

### 3.1 コンポーネント一覧

| コンポーネント名 | AWS サービス | 役割と責任 |
|--------------|------------|---------|
| APIアプリケーション | ECS Fargate | Spring Bootアプリケーションの実行、APIエンドポイントの提供 |
| ロードバランサー | Application Load Balancer (内部) | APIリクエストの負荷分散、ヘルスチェック |
| データベース | RDS for PostgreSQL | 確認記録データの永続化、トランザクション管理 |
| ネットワーク | VPC, Subnet, Security Group | セキュアな通信経路の提供、アクセス制御 |
| 監視 | CloudWatch, X-Ray | メトリクス収集、ログ管理、パフォーマンストレース |
| 通知 | SNS | アラート通知、障害エスカレーション |
| CI/CD | GitHub Actions | 継続的インテグレーション・デリバリーの自動化 |
| ルーティング | VPC, Route Table | ネットワーク内のトラフィック制御 |
| アクセス管理 | IAM | サービス間認証、最小権限の実現 |

### 3.2 コンポーネント間の関係性

```
[APIクライアント] --> [ALB] --> [ECS Fargate] --> [RDS PostgreSQL]
                                    |
                                    v
                                [CloudWatch]
                                    |
                                    v
                                 [SNS]
                                    |
                                    v
                             [システム管理者]
```

- **APIクライアント → ALB**: APIリクエストの送信（JWT認証付き）
- **ALB → ECS Fargate**: リクエストのルーティングとロードバランシング
- **ECS Fargate → RDS**: データの永続化、参照
- **ECS Fargate → CloudWatch**: ログ出力、メトリクス送信
- **CloudWatch → SNS**: アラート通知
- **SNS → システム管理者**: Eメール/SMS通知

## 4. データフロー図

### 4.1 API呼び出しフロー

```
┌──────────────┐      ┌───────────┐      ┌─────────────┐      ┌──────────────┐
│              │      │           │      │             │      │              │
│ APIクライアント │ ──────> │ ALB (内部) │ ──────> │ ECS Fargate │ ──────> │ RDS PostgreSQL │
│              │      │           │      │             │      │              │
└──────────────┘      └───────────┘      └─────────────┘      └──────────────┘
       │                                        │
       │                                        │
       │                                        v
       │                                 ┌─────────────┐
       │                                 │             │
       └───────────────────────────────> │ CloudWatch  │
                                         │ Logs/Metrics│
                                         │             │
                                         └─────────────┘
```

1. APIクライアントが認証トークン（JWT）付きリクエストを送信
2. ALBがリクエストを受け取り、ヘルスチェックに基づいて適切なECS Fargateタスクにルーティング
3. ECS Fargateタスク上のSpring Bootアプリケーションがリクエストを処理
4. アプリケーションがRDS PostgreSQLにデータアクセス（必要に応じて）
5. アプリケーションがレスポンスを生成し、ALB経由でクライアントに返却
6. オペレーションのログとメトリクスがCloudWatchに送信される

### 4.2 デプロイフロー

```
┌──────────┐      ┌───────────┐      ┌─────────────┐      ┌───────────────┐
│          │      │           │      │             │      │               │

│          │      │ Repository │      │ Actions    │      │               │
└──────────┘      └───────────┘      └─────────────┘      └───────────────┘
                                            │                      │
                                            │                      │
                                            v                      v
                                     ┌─────────────┐       ┌───────────────┐
                                     │             │       │               │
                                     │ ECR         │ ─────> │ ECS Fargate    │
                                     │             │       │               │
                                     └─────────────┘       └───────────────┘
```

1. 開発者がコードを変更し、GitHubリポジトリにプッシュ
2. GitHub Actionsが自動的にビルドとテストを実行
3. テスト通過後、DockerイメージがビルドされてECRにプッシュ
4. 同時に、CloudFormationテンプレートが検証され、AWSリソースが更新
5. ECS Fargateサービスが新しいイメージを使用して更新される

## 5. 非機能要件の実現方法

### 5.1 可用性の実現

- **マルチAZ構成**: 2つのアベイラビリティゾーンにまたがるデプロイ
- **自動ヘルスチェック**: ALBによるECSタスクのヘルスチェックと異常インスタンスの除外
- **自動復旧**: ECSタスクの異常時自動再起動
- **RDSマルチAZ**: スタンバイレプリカによる自動フェイルオーバー

### 5.2 セキュリティの実現

- **プライベートサブネット**: すべてのコンポーネントをプライベートサブネット内に配置
- **セキュリティグループ**: 必要最小限のポートとプロトコルのみを許可
- **JWT認証**: APIリクエストの認証に安全なトークンベース認証を採用
- **暗号化**: 保存データと転送中データの暗号化
- **IAM最小権限**: サービスごとに必要最小限の権限のみを付与

### 5.3 スケーラビリティの実現

- **ECS Auto Scaling**: CPU使用率に基づく自動スケーリング設定
- **ALBによる負荷分散**: 複数のECSタスク間での負荷分散
- **RDSインスタンスタイプ**: 環境ごとに適切なサイジング

### 5.4 運用性の実現

- **CloudWatch Logs**: 集中型ログ管理
- **CloudWatch Metrics**: パフォーマンスメトリクスの収集と可視化
- **CloudWatch Alarms**: 異常検知と通知
- **X-Ray**: 分散トレーシングによるパフォーマンス分析
- **SNS通知**: 重要アラートのリアルタイム通知

## 6. 環境分離戦略

### 6.1 環境構成

システムは以下の3つの環境を構築します：

- **開発環境（dev）**
  - 目的: 開発者の日常的な開発・テスト用
  - 特徴: 最小リソース、柔軟な設定、自動テスト統合

- **ステージング環境（stg）**
  - 目的: 本番リリース前の最終検証用
  - 特徴: 本番相当構成、性能テスト環境、リリース前検証

- **本番環境（prd）**
  - 目的: 実サービス提供用
  - 特徴: 高可用性構成、適切なリソースサイジング、厳格なセキュリティ設定

### 6.2 環境差分

| 機能/リソース | 開発環境（dev） | ステージング環境（stg） | 本番環境（prd） |
|-------------|--------------|-------------------|--------------|
| VPC/ネットワーク | 単一AZ | マルチAZ（2AZ） | マルチAZ（2AZ） |
| ECS Fargate | 最小1タスク、最大2タスク | 最小2タスク、最大5タスク | 最小2タスク、最大10タスク |
| RDSインスタンス | db.t3.small, シングルAZ | db.t3.medium, マルチAZ | db.t3.large, マルチAZ |
| ALBタイプ | 内部 | 内部 | 内部 |
| アラート通知 | 重大な問題のみ、メールのみ | すべての問題、メールのみ | すべての問題、メール+SMS |
| ログ保持期間 | 14日間 | 30日間 | 90日間 |
| バックアップ | 自動バックアップ（7日間） | 自動バックアップ（7日間）+ 週次スナップショット | 自動バックアップ（7日間）+ 週次スナップショット + 月次スナップショット（1年） |

## 7. IaC戦略

### 7.1 IaC管理方法

- **テンプレート形式**: AWS CloudFormation（YAML形式）
- **ディレクトリ構造**: 
  ```
  /iac/cloudformation/
    ├── templates/         # サービス別テンプレート
    │   ├── vpc.yaml       # ネットワーク構成
    │   ├── rds.yaml       # データベース
    │   ├── ecs.yaml       # ECSクラスタとサービス
    │   ├── alb.yaml       # ロードバランサー
    │   └── monitoring.yaml # 監視・アラート
    ├── config/            # 環境別設定
    │   ├── dev/           # 開発環境
    │   │   └── parameters.json
    │   ├── stg/           # ステージング環境
    │   │   └── parameters.json
    │   └── prd/           # 本番環境
    │       └── parameters.json
    └── scripts/           # デプロイスクリプト
        ├── deploy.sh
        └── validate.sh
  ```

### 7.2 パラメータ戦略

CloudFormationテンプレートは、環境ごとに異なるパラメータ値を使用して同一のテンプレートを適用します。

- **共通パラメータ**:
  - プロジェクト名
  - サービス名
  - タグ設定

- **環境別パラメータ**:
  - VPC CIDR範囲
  - サブネット構成
  - インスタンスタイプ（RDS、Fargate）
  - Auto Scaling設定
  - アラーム閾値
  - バックアップ設定

## 8. セキュリティ設計原則

### 8.1 ネットワークセキュリティ

- プライベートサブネット内にすべてのサービスを配置
- セキュリティグループによるトラフィック制御
- NACLによる追加のネットワークフィルタリング
- VPC外部からのアクセスを内部ALBに限定

### 8.2 認証・認可

- JWT（JSON Web Token）を使用したAPIアクセス認証
- IAMロールによるAWSサービス間認証
- 最小権限の原則に基づくポリシー設定

### 8.3 暗号化

- 転送中データのTLS 1.2+による暗号化
- 保管データのAWS KMSを使用した暗号化
- RDSデータベースの保存時暗号化

### 8.4 監査とコンプライアンス

- CloudTrailによるAPI呼び出しの記録
- CloudWatchによるログとメトリクスの集中管理
- AWS Config準拠ルールによる設定モニタリング

## 9. 今後の拡張性

本アーキテクチャは、将来の拡張を考慮して以下の対応が可能です：

1. **APIの機能拡張**: 新しいエンドポイントの追加が容易
2. **トラフィック増加**: ECS Auto Scaling設定の調整により対応可能
3. **データ量増加**: RDSインスタンスサイズのアップグレードや読み取りレプリカの追加
4. **他サービスとの連携**: イベント駆動アーキテクチャへの発展（EventBridge, SNS, SQS等の追加）
5. **マルチリージョン展開**: 地理的冗長性のための拡張（将来課題）

## 10. 設計レビュー

本アーキテクチャ設計は、AWSのWell-Architected Frameworkの5つの柱に基づきレビューされます：

1. **運用上の優秀性**: IaCによる自動化、モニタリング、アラート体制
2. **セキュリティ**: 多層防御、最小権限、暗号化
3. **信頼性**: マルチAZ、自動復旧、バックアップ戦略
4. **パフォーマンス効率**: 適切なサービス選択、Auto Scaling
5. **コスト最適化**: 環境ごとのリソースサイジング、必要時のみリソース利用
